# 0.2.0 design — Podman, Create, Edit

Design for the three 0.2.0 features. Written to implement from; read it before
starting any of the three. Verified against the vendored bollard 0.21 /
bollard-stubs source we compile against (CLAUDE.md rule 1), not from memory.

## The principle that shapes "Edit": containers are immutable

Docker's `POST /containers/{id}/update` — bollard `update_container` — changes
**only** runtime resource limits and the restart policy (`memory`, `nano_cpus`,
`cpu_quota`, `cpuset_cpus`, `pids_limit`, `restart_policy`, …). It **cannot**
change image, ports, env, volumes, command, or network. `rename_container`
changes the name. Everything else is frozen at create time.

So there is no in-place edit for the interesting fields. To change image / ports
/ env you **recreate**: read the old config, stop + remove the old container,
create a new one with the modified config. This is exactly how Docker Desktop
and Portainer "edit" — there is no other option. The practical consequence:
**Edit is mostly Create with a pre-filled form**, so Create is built first and
Edit reuses it.

## Recorded scope decisions

- **Create** ships **lean**: image, name, ports, env. Volumes, restart policy,
  resource limits, and command override are deferred (the mapping and form are
  designed so they slot in without rework).
- **Edit** covers **both** tiers: live edits (rename + resources/restart, no
  downtime) **and** recreate (image/ports/env, with a warning).
- **Podman** is auto-detected alongside Docker; no manual switch required for v1.
- No new crates. bollard already carries create / pull / update / rename; the
  form is plain GTK + a relm4 factory.

## Feature A — Podman support

**Feasibility: high, small.** bollard is Docker *and* Podman first-class, and
Podman exposes a Docker-compatible socket. Once connected, every call we already
make (`list_containers`, `start`/`stop`, `logs`, `stats`, and the new create /
update calls) works unchanged — same API. The work is confined to socket
discovery in `client.rs`.

### Socket discovery (extend `resolve_endpoint`, keep it one function — rule 2)

bollard ships `Docker::connect_with_podman_defaults()`, but it is *Podman-first*
and hides discovery. We keep our own `resolve_endpoint` because it already owns
tailored, actionable diagnostics (the `EACCES`/`docker` group message) and a
Docker-first default. We just add the Podman paths:

```
1. DOCKER_HOST                                   (unchanged; a URL, hand to bollard)
2. rootless: $XDG_RUNTIME_DIR/docker.sock
             $XDG_RUNTIME_DIR/podman/podman.sock
3. rootful:  /var/run/docker.sock
             /run/podman/podman.sock
```

Docker wins when both are present (backward compatible). The `Endpoint` enum
gains a runtime tag so `diagnose` can tailor the remedy:

```rust
enum Runtime { Docker, Podman }
enum Endpoint {
    DockerHost(String),
    Socket { path: PathBuf, runtime: Runtime },
}
```

`connect` is unchanged (`connect_with_socket` speaks the Docker API to either).
`diagnose` branches on `runtime` for the "socket present but daemon silent" and
"nothing listening" cases:

- Docker: `sudo systemctl start docker.socket`
- Podman: `systemctl --user enable --now podman.socket` (rootless) /
  `sudo systemctl enable --now podman.socket` (rootful)

### Gotchas / testing

- Podman's socket is **not enabled by default** — the likeliest failure is "no
  socket", and the remedy text above is the whole point.
- The API is Docker-compatible but not identical; a handful of endpoints differ
  in detail. **Test list / create / logs / stats against a real rootless
  Podman** before calling this done. Rootless Podman is the common case.
- Optional (later): a Preferences toggle "Runtime: Auto / Docker / Podman" if
  auto-detection ever guesses wrong. Not needed for v1.

**Effort: small.** No new modules, no new types, no UI.

## Feature B — Create (lean: image, name, ports, env)

### Our type (rule 3 — bollard types never reach the form)

```rust
pub struct ContainerSpec {
    pub image: String,             // "nginx:1.27"; no tag => Docker defaults to :latest
    pub name: Option<String>,      // None => Docker assigns a random name
    pub ports: Vec<PortMapping>,   // published ports only
    pub env: Vec<EnvVar>,
    // deferred, but leave room: volumes, restart, mem/cpu limits, command
}
pub struct PortMapping { pub host: u16, pub container: u16, pub protocol: Protocol } // reuse existing Protocol
pub struct EnvVar { pub key: String, pub value: String }
```

### Mapping `ContainerSpec` -> `ContainerCreateBody` (pure fn, unit-tested)

Confirmed field shapes: `ContainerCreateBody.env: Option<Vec<String>>` (`"K=V"`),
`.exposed_ports: Option<Vec<String>>` (`"80/tcp"`), `.host_config.port_bindings:
Option<PortMap>` where `PortMap = HashMap<String, Option<Vec<PortBinding>>>`,
`PortBinding { host_ip: Option<String>, host_port: Option<String> }`.

```rust
fn to_create_body(spec: &ContainerSpec) -> ContainerCreateBody {
    let key = |p: &PortMapping| format!("{}/{}", p.container, proto_str(p.protocol)); // "80/tcp"

    let exposed: Vec<String> = spec.ports.iter().map(key).collect();

    let mut bindings: PortMap = HashMap::new();
    for p in &spec.ports {
        bindings.entry(key(p)).or_insert_with(|| Some(Vec::new()))
            .get_or_insert_with(Vec::new)
            .push(PortBinding { host_ip: None, host_port: Some(p.host.to_string()) });
    }

    ContainerCreateBody {
        image: Some(spec.image.clone()),
        env: Some(spec.env.iter().map(|e| format!("{}={}", e.key, e.value)).collect()),
        exposed_ports: (!exposed.is_empty()).then_some(exposed),
        host_config: Some(HostConfig {
            port_bindings: (!bindings.is_empty()).then_some(bindings),
            ..Default::default()   // restart_policy/memory/nano_cpus slot in here later
        }),
        ..Default::default()
    }
}
```

Name rides in the query options, not the body:
`CreateContainerOptionsBuilder::default().name(n).build()` (confirm the exact
setter against `bollard::query_parameters` at build time — the builders are
generated, like the `ListContainersOptionsBuilder` we already use).

### `client.rs` additions

```rust
// Pull an image, streaming progress. CreateImageOptions takes from_image + tag.
pub fn pull_image<'a>(docker: &'a Docker, image: &'a str)
    -> impl Stream<Item = Result<PullProgress, String>> + Send + 'a;

pub async fn create_container(docker: &Docker, spec: &ContainerSpec) -> Result<String>; // returns id

// Orchestrator: ensure image, create, start.
pub async fn create_and_start(docker: &Docker, spec: &ContainerSpec) -> Result<String>;
```

`create_and_start` flow, and how "image missing" is handled:

```
1. create_container(spec)
   - Ok(id) -> go to 3
   - Err(404 "No such image") -> pull_image(spec.image) to completion, then create again
   - Err(other) -> surface
2. (image pulled) create_container(spec) -> id
3. start_container(id)
```

**MVP pull UX:** drive `pull_image` to completion behind an indeterminate
spinner in the dialog (no per-layer bar). The stream is kept so a later version
can parse `CreateImageInfo` (status / progress_detail) into a real progress bar
without changing the client signature.

### Image discovery (decided: local images + daemon search, no new dependency)

A blank text field is a poor way to pick an image. Two mechanisms, both through
the daemon we're already connected to — no HTTP client, no new crate, and
registry-agnostic (it respects whatever registries the runtime is configured
for, which matters because we already run non-Hub images like
`lscr.io/linuxserver/*`).

**Why not the Docker Hub HTTP API.** Docker Hub's v2 endpoints are open and
no-auth (verified): `…/v2/search/repositories/` adds pull counts, and
`…/v2/repositories/{ns}/{repo}/tags/` would give a real tag dropdown. But
reaching them needs an HTTP-client dependency (bollard's transport is bound to
the daemon socket, not arbitrary HTTPS) and only ever covers Docker Hub. Deferred
as an optional Phase 2 (a tag picker + pull-count sort) behind an explicit
dependency decision — not in v1.

**1. Local images — the fast path.** `bollard::list_images` →
`ImageSummary.repo_tags` gives ready-to-run `"repo:tag"` strings. Offered as
`GtkEntryCompletion` on the image field. Offline, instant, and it's the common
case ("run my postgres again"). Drop the dangling `"<none>:<none>"` entries.

**2. Daemon search — for something new.** A search affordance
(`system-search-symbolic` suffix on the image row, or a small "Find image"
`adw::Dialog`) runs `bollard::search_images` and lists results (name, ⭐ stars,
an "official" badge, description). Picking one fills the image field; since
search returns no tags, the tag defaults to `:latest` and stays editable.

New types (rule 3 — the bollard search item never reaches the view):

```rust
pub struct ImageSearchResult {
    pub name: String,        // "nginx", "nginx/nginx-ingress"
    pub description: String,
    pub stars: u64,
    pub official: bool,      // is_automated is dropped — deprecated, always false now
}
```

New `client.rs` functions (both map bollard → our types at the boundary, pure and
unit-testable):

```rust
pub async fn list_local_images(docker: &Docker) -> Result<Vec<String>>;          // "repo:tag", minus <none>
pub async fn search_images(docker: &Docker, term: &str) -> Result<Vec<ImageSearchResult>>;
```

`ImageSearchResponseItem` carries `name`/`description`/`star_count`/`is_official`
(confirmed); `search_images` takes a `SearchImagesOptionsBuilder` (term + an
optional limit). The search UI can be a small dedicated component
(`components/image_search.rs`) or live inside the form — decide at build time;
either way it returns the chosen `"repo:tag"` back to the form.

**Podman note:** Podman implements `/images/search` and searches every registry
in `registries.conf`, so this is genuinely cross-registry there. Caveat: some
registries don't implement the search endpoint at all, so results can be empty
for a registry that's reachable for `pull` — the field must still accept a
typed reference, always.

### Form component — `components/container_form.rs`

A relm4 `Component` presented as an `adw::Dialog` (consistent with
About/Preferences/Shortcuts). Body is an `adw::PreferencesPage` with groups:

```
Image        [ adw::EntryRow ]                         (required)
Name         [ adw::EntryRow ]                         (optional)
Ports        <FactoryVecDeque<PortRow>>  + "Add port"  (host | container | proto)
Environment  <FactoryVecDeque<EnvRow>>   + "Add variable" (key | value)
             [ Cancel ]                 [ Create ]
```

- **Repeatable rows** are `FactoryComponent`s in `components/form_rows.rs` (a
  `PortRow` and an `EnvRow`), each owning its fields + a remove `⋮`/`✕`. This is
  the same factory pattern as the container list — the idiomatic relm4 way to do
  add/remove lists, and it keeps per-row widget state where it belongs.
- **Model** holds the in-progress `ContainerSpec` plus a validity flag; the
  Create button binds `sensitive` to validity (`#[watch]`).
- **Messages:** `ImageChanged`, `NameChanged`, `AddPort`, `AddEnv`, row outputs
  (`PortChanged{index,..}`, `RemovePort(index)`, …), `Submit`, `Cancel`.
- **Output:** `FormOutput::Submit(ContainerSpec)` / `FormOutput::Cancel`.
- Same component serves Edit — see Feature C. An `init` param decides Create vs
  Edit (title/button label, and whether it's seeded from a spec).

### Validation (in the form model; surface inline, don't rely on the daemon)

- image: non-empty (a blank image is the one thing that trivially fails).
- host/container ports: parse to `1..=65535`; flag duplicates on the same
  host port + protocol.
- env keys: non-empty, no `=` in the key.
- name (if set): matches `^[a-zA-Z0-9][a-zA-Z0-9_.-]+$` (Docker's rule) — but a
  name collision (409) can only be caught by the daemon, so still map that error
  to a clear toast.

### `app.rs` wiring

- A **New** button in the header (`list-add-symbolic`), left of the search
  toggle or in the primary menu. Opens the form.
- `AppMsg::NewContainer` builds + presents the form; its `Submit` forwards to
  `AppMsg::CreateContainer(ContainerSpec)`.
- `CreateContainer` dispatches a `command` (pull may stream) →
  `CommandMsg::ContainerCreated { name, result }`.
- On success: toast "Created {name}", and the existing 2 s poll shows the new
  row (or `sender.input(Refresh)` to show it immediately, like `ActionDone`).
- Failures → toast via the existing `short_reason` path (extend it if create/pull
  surface new 4xx shapes worth special-casing, e.g. 409 name conflict).

### Tests

- `to_create_body`: ports → both `exposed_ports` and `port_bindings`, env → `K=V`,
  name empty vs set, protocol suffixes, dedup — pure, exactly like the existing
  `types.rs` mapping tests.
- Validation predicates (port range, dup detection, name regex).
- Image discovery mappers: `ImageSearchResponseItem` → `ImageSearchResult`, and
  `ImageSummary.repo_tags` → the local-image list with `<none>:<none>` dropped.

**Effort: large** — the factory rows + validation are the bulk. Everything else
is a straight application of patterns already in the repo.

## Feature C — Edit

Two independent pieces. Tier A can ship without Create; Tier B reuses it.

### Tier A — live edits (rename + resources), no downtime

What `update_container` / `rename_container` actually allow, surfaced as a small
"Edit…" dialog from the row `⋮` menu and the detail page:

```rust
pub struct ResourceEdit {
    pub name: Option<String>,          // -> rename_container (only if changed)
    pub memory_bytes: Option<i64>,     // 0/None = unlimited
    pub nano_cpus: Option<i64>,        // cpus * 1e9
    pub restart: RestartPolicyName,    // No | Always | UnlessStopped | OnFailure
}
```

- rename: `rename_container(id, RenameContainerOptions{ name })` — only when the
  name changed.
- resources + restart: `update_container(id, ContainerUpdateBody { memory,
  nano_cpus, restart_policy: Some(RestartPolicy{ name: Some(<enum>), .. }), ..default })`.
  `RestartPolicyNameEnum` = NO | ALWAYS | UNLESS_STOPPED | ON_FAILURE.
- Works on running containers, no restart, no data loss. Small, safe, satisfying.

### Tier B — recreate (image / ports / env), reuses the Create form

**Inverse mapping** (new pure fn, unit-tested): `ContainerInspectResponse ->
ContainerSpec`.

```rust
fn spec_from_inspect(resp: &ContainerInspectResponse) -> ContainerSpec
// image  <- config.image
// name   <- resp.name (strip leading '/')
// env    <- config.env, split each on the first '=' (values may contain '=')
// ports  <- host_config.port_bindings (same shape ports_from_inspect already parses)
```

The form opens in **Edit mode** seeded from this spec. On submit, orchestrate a
**safe recreate with rollback** — never leave the user with nothing:

```
1. inspect old -> spec (+ was_running)          [seed the form]
2. user edits, submits new spec
3. stop old (if running)
4. rename old -> "<name>-old-<shortid>"          [frees the name]     rename_container
5. create new (spec, name = <name>) + start
   - success -> remove old                        [remove_container]
   - failure -> rename old back to <name>,         [rollback]
                restart if was_running, surface the error
```

Renaming the old container out of the way first (step 4) is what makes step 5
safe: if create fails, the original still exists and is restored. Warn up front,
because even on success:

- **anonymous-volume data is not carried over** (named volumes/binds are, since
  they're in the recreated config); the container **id changes**; there is a
  **brief downtime**.

Present that in the confirm step (an `adw::AlertDialog`, like Remove).

> A near-free future variant is **Duplicate**: run steps 1–2 and 5 only (create a
> new container from the pre-filled spec, leave the original alone). Same form,
> no destructive steps. Out of v1 scope but noted — it's the recreate flow minus
> the risk.

### `client.rs` additions

```rust
pub async fn rename_container(docker: &Docker, id: &str, name: &str) -> Result<()>;
pub async fn update_resources(docker: &Docker, id: &str, edit: &ResourceEdit) -> Result<()>;
// Tier B is orchestration over inspect/stop/rename/create/start/remove; it can
// live in app.rs as a command sequence, or as a client::recreate helper.
```

### `app.rs` wiring

- Row `⋮` and detail page gain **Edit…** (Tier A dialog) and **Edit config…**
  (Tier B → the Create form in edit mode). Or one "Edit" entry that opens a form
  with both the resource fields and the recreate fields, and decides per changed
  field whether a live update suffices or a recreate is required — cleaner UX,
  more logic. Decide at implementation time; the simple two-entry version is the
  smaller first step.

### Tests

- `spec_from_inspect`: image/name/env/ports round-trip against a
  `ContainerInspectResponse` fixture (mirrors the existing `maps_published_ports_from_inspect`
  test), including env values that themselves contain `=`.
- `to_create_body(spec_from_inspect(x))` round-trips the fields we model.

**Effort:** Tier A small; Tier B medium (inverse mapping + orchestration +
warnings), given Create already exists.

## Proposed implementation milestones

Independent, each its own PR, each green under `clippy -D warnings` + tests:

- **M1 — Podman** (`client.rs` only): socket paths + `Runtime` tag + `diagnose`
  text. Test against real rootless Podman. Smallest, no UI, unblocks testing the
  rest against Podman.
- **M2 — Create MVP**: `ContainerSpec` + `to_create_body` + `pull_image` /
  `create_and_start` + image discovery (`list_local_images` completion +
  `search_images` picker) + `container_form.rs` + `form_rows.rs` + header button
  and wiring. The big one. (Optional Phase 2, separate: Docker Hub HTTP tag
  picker + pull-count sort — needs a dependency decision.)
- **M3 — Edit / live** (Tier A): `ResourceEdit` + `rename_container` /
  `update_resources` + a small dialog. Independent of M2.
- **M4 — Edit / recreate** (Tier B): `spec_from_inspect` + the rollback-safe
  recreate orchestration + confirm/warn, reusing the M2 form in edit mode.

## Open questions / risks

1. **Where does `ContainerSpec` live** — extend `types.rs` or a new
   `docker/spec.rs`? (Cohesion vs file size.)
2. **One "Edit" or two?** A single smart Edit (auto-choosing live-update vs
   recreate per changed field) is nicer UX but more logic than two explicit
   entries. M3/M4 can start with two and merge later.
3. **Scope drift (flagging, per CLAUDE.md).** Create/recreate pull ports, env,
   and eventually volumes/networks into the app — several are on the "stay lean"
   list. This is the largest single expansion of Dockyard's surface toward
   Docker-Desktop territory. Deliberate and wanted; recorded here so it stays a
   conscious choice.
4. **Podman parity** — a couple of endpoints differ; M1's definition of done is
   "list/create/logs/stats verified on real Podman", not "it compiled".
5. **Phase-2 image discovery** — the Docker Hub HTTP tag picker + pull-count sort
   is genuinely nicer UX but costs an HTTP-client dependency and is Hub-only.
   Deferred; revisit once Create is in use and we know whether typing the tag is
   actually a pain point.
